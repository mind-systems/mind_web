## Phase 2 — Authentication

- [x] **Auth context + token management** — Create `src/core/auth/AuthContext.tsx`. Context value: `{ token: string | null; login(token: string): void; logout(): void; pendingEmail: string | null; setPendingEmail(email: string): void; clearPendingEmail(): void }`. `token` persisted under `mind_auth_token`; `pendingEmail` under `mind_pending_email` — both in `localStorage`, both managed exclusively by `AuthContext` (no other file may read or write these keys). `login()` stores token and navigates to `/sessions`. `logout()` clears token and navigates to `/login`. Export `useAuth()` hook. `ProtectedRoute` reads from `useAuth().token`. Task ends when `useAuth().logout()` from a protected page redirects to `/login`. [13m 24s]

- [x] **Login page: email OTP flow** — Implement `LoginPage`. Step 1: email input + "Send code" button → `POST /auth/send-code { email }`. On success: call `auth.setPendingEmail(email)` (persists under `mind_pending_email` via AuthContext — do not write to localStorage directly), advance to step 2. Step 2: 6-digit code input + "Verify" button → `POST /auth/verify-code { email, code }` returns `{ accessToken, user }` → `auth.login(accessToken)`. Show loading spinner during requests. Show inline error on failure (extract `message` field from response body). Centered card, max-width 400px. Note: these REST endpoints live in mind_api Phase 21. [9m 12s]

- [x] **Magic link auto-login** — Implement `MagicLinkPage` at `/deeplink-auth`. The API email template already contains `{APP_BASE_URL}/deeplink-auth?code=XXXXXX` — on mobile this opens as a native deeplink, in a browser this hits this route. On mount: read `?code=` from URL query params. Read `auth.pendingEmail` (set by `LoginPage` via `auth.setPendingEmail()`). If both present: call `POST /auth/verify-code { email, code }` → `auth.login(token)` → navigate to `/sessions`. If code is present but `auth.pendingEmail` is null: show a short email input so the user can confirm their address, then verify. Call `auth.clearPendingEmail()` after successful login. No changes to API email templates needed — the existing deeplink URL works for web as-is. [5m 58s]

- [x] **Login page: Google Sign-In** — Add "Continue with Google" button on `LoginPage`. On click: call `GET /auth/google` → API redirects browser to Google OAuth. After Google auth, API relays to `{WEB_BASE_URL}/auth/google/callback?googleCode=<code>` (success) or `?googleError=<err>` (failure). Implement `GoogleCallbackPage`: on `googleCode` → call `POST /auth/google { code, redirectUri: window.location.origin + '/auth/google/callback' }` → receive `{ accessToken, user }` → `auth.login(accessToken)`. On `googleError` → redirect to `/login?error=google`. Note: endpoint spec in mind_api Phase 21 (`.ai-factory/ROADMAP.md`). [14m 24s]

- [x] **API client with auth interceptor** — Create `src/core/api/client.ts`. `apiFetch<T>(path, options?)`: prepend `API_BASE_URL`, add `Content-Type: application/json` and `Authorization: Bearer <token>` headers. On 401: clear token, navigate to `/login`. On non-2xx: throw `ApiError(status, message)` with message extracted from response JSON. Task ends with the client used by at least one page. [6m 54s]

## Phase 8 — Security

- [x] **OAuth `state` CSRF protection (SPA side)** — The Google sign-in relay carries no `state`, leaving it open to login-CSRF: an attacker's `code`, fed to the victim's callback, silently logs the victim into the attacker's account. Today `LoginPage.handleGoogleSignIn` does `window.location.href = ${API_BASE_URL}/auth/google` with no state, and `GoogleCallbackPage` exchanges `googleCode` with no state check. The SPA is the OAuth client and owns `state` (backend holds no session). (1) Add `src/core/auth/oauthState.ts` with `createOAuthState()` (generate a `crypto.randomUUID()` opaque token, store under `sessionStorage` key `oauth_state`, return it) and `consumeOAuthState()` (read-and-remove, one-time use); use `sessionStorage` (tab-scoped), not `localStorage`, and keep this helper the only place that touches it so no page reads storage directly. Note on the project rule: `CLAUDE.md`'s storage rule is a closed allow-list — `localStorage` access only in `core/auth/AuthContext.tsx` and `core/api/client.ts` — and is specifically about `localStorage`, so a new `sessionStorage` helper does not break it literally; to honor its spirit (storage centralized in dedicated modules), at implementation time either add `oauthState.ts` to that allow-list in `CLAUDE.md` (clarifying it also covers `sessionStorage`) or put the two helpers directly in `AuthContext.tsx`. (2) `LoginPage.handleGoogleSignIn`: `const state = createOAuthState();` then redirect to `${API_BASE_URL}/auth/google?state=${encodeURIComponent(state)}`. (3) `GoogleCallbackPage`: after the existing `didExchange` ref guard (so StrictMode's double-invoke does not consume the one-time state twice), read `state` from the query and compare to `consumeOAuthState()` **before** anything else — if missing or mismatched, `navigate('/login?error=google', { replace: true })` and do NOT call `POST /auth/google`; if it matches, proceed with the existing `googleCode`/`googleError` handling and code exchange. Optionally include `state` in the `POST /auth/google` body (backend logs it, does not validate). Ships in lockstep with mind_api Phase 26 (backend relays `state` through `GET /auth/google` and the callback, and accepts it unvalidated on `POST /auth/google`) — neither half does anything until both ship. Full spec: `notes/13-oauth-state-csrf-requirements.md`. [5m 16s]
